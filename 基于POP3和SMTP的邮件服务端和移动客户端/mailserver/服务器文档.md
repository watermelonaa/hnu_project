# 1.引言

## 1.1 编写目的

本文档用于明确邮件系统服务器端项目的具体需求，包括SMTP服务器、POP3服务器和Web管理后台的功能需求、性能需求以及其他非功能性需求，使其成为服务器端开发、测试和维护的主要依据。

## 1.2 背景

项目选择的背景依托计算机网络课程教学中对SMTP与POP3协议的讲述内容。旨在开发一款基于PHP的邮件服务器系统，实现完整的SMTP和POP3协议支持，并提供Web管理界面。

该邮件服务器系统的参与者包括：发件人、收件人、管理员。根据该邮件系统的参与者，将其划分为以下模块：SMTP模块、POP3模块、用户管理模块、管理员模块。

## 1.3 问题分析

在当前市面现有的邮件服务器系统中往往会存在以下问题：

1. 配置复杂，需要深入理解邮件协议才能部署和维护。
2. 缺乏友好的管理界面，需要通过命令行或配置文件进行管理。
3. 功能单一，缺乏灵活的过滤和管理机制。

针对以上问题，我们期望设计一款轻量、简单、易用的邮件服务器系统。既能提供完整的SMTP和POP3协议支持，又能通过Web界面进行便捷的管理和配置。

# 3.服务器端功能性需求分析

## 3.1 SMTP服务器

SMTP服务器负责接收客户端发送的邮件，并进行存储和管理。

### 3.1.1 邮件接收

**基本流程：**

SMTP服务器监听25端口（可配置），等待客户端连接。当客户端连接后，服务器按照SMTP协议流程处理邮件发送请求：

1. 服务器发送欢迎消息："220 mail.simple.com SMTP Ready"
2. 客户端发送HELO/EHLO命令，服务器响应"250 OK - Hello"
3. 客户端发送MAIL FROM命令指定发件人，服务器验证并响应"250 Sender OK"
4. 客户端发送RCPT TO命令指定收件人（支持多个收件人），服务器验证并响应"250 Recipient OK"
5. 客户端发送DATA命令，服务器响应"354 Start mail input, end with <CRLF>.<CRLF>"
6. 客户端发送邮件内容，以单独一行的"."结束
7. 服务器接收邮件内容，解析主题和正文，保存到数据库
8. 服务器响应"250 Mail accepted"
9. 客户端发送QUIT命令，服务器响应"221 Bye"并关闭连接

**分支流程：**

1. 如果客户端IP被过滤规则阻止，服务器响应"550 IP address blocked"并断开连接
2. 如果发件人邮箱被过滤规则阻止，服务器响应"550 Sender email blocked"并断开连接
3. 如果收件人邮箱被过滤规则阻止，服务器响应"550 Recipient email blocked"，但继续处理其他收件人
4. 如果收件人邮箱容量已满，服务器响应"552 Mailbox full"，跳过该收件人
5. 如果所有收件人都无效，服务器响应"503 No valid recipients"

**数据需求：**

1. 发件人邮箱地址
2. 收件人邮箱地址（支持多个）
3. 邮件主题
4. 邮件正文内容
5. 邮件大小（字节）
6. 客户端IP地址
7. 邮件接收时间

### 3.1.2 邮件过滤

**基本流程：**

SMTP服务器在处理邮件时，会检查以下过滤规则：

1. **IP地址过滤**：在HELO命令后检查客户端IP是否在黑名单中
2. **发件人邮箱过滤**：在MAIL FROM命令后检查发件人邮箱是否被阻止
3. **收件人邮箱过滤**：在RCPT TO命令后检查收件人邮箱是否被阻止

**分支流程：**

1. 如果IP被阻止，立即断开连接
2. 如果发件人被阻止，拒绝该邮件
3. 如果收件人被阻止，跳过该收件人，继续处理其他收件人

**数据需求：**

1. 过滤规则类型（email/ip）
2. 过滤规则值
3. 过滤动作（block/allow）
4. 规则状态（激活/禁用）

### 3.1.3 邮箱容量管理

**基本流程：**

SMTP服务器在接收邮件时，会检查收件人的邮箱容量限制：

1. 在RCPT TO阶段进行初步检查，估算邮件大小（默认50KB）
2. 在接收完整邮件内容后，使用实际邮件大小进行最终检查
3. 如果收件人邮箱已满，拒绝该邮件并记录日志

**分支流程：**

1. 如果邮箱使用量 + 邮件大小 > 邮箱容量限制，拒绝该邮件
2. 如果所有收件人邮箱都已满，返回错误信息

**数据需求：**

1. 用户邮箱容量限制（默认100MB）
2. 用户当前邮箱使用量
3. 邮件实际大小

## 3.2 POP3服务器

POP3服务器负责提供邮件检索和下载服务，允许客户端通过POP3协议获取邮件。

### 3.2.1 用户认证

**基本流程：**

POP3服务器监听110端口（可配置），等待客户端连接。认证流程如下：

1. 服务器发送欢迎消息："+OK POP3 Simple Server Ready"
2. 客户端发送USER命令指定用户名，服务器验证用户是否存在且激活
3. 如果用户存在，服务器响应"+OK User found"
4. 客户端发送PASS命令指定密码，服务器验证密码
5. 如果密码正确，服务器加载用户邮件列表，响应"+OK Logged in, {count} messages"
6. 进入事务阶段（TRANSACTION）

**分支流程：**

1. 如果用户不存在或未激活，服务器响应"-ERR User not found"
2. 如果密码错误，服务器响应"-ERR Invalid password"
3. 如果未先发送USER命令就发送PASS，服务器响应"-ERR USER first"

**数据需求：**

1. 用户名（邮箱地址）
2. 密码（加密存储）
3. 用户激活状态

### 3.2.2 邮件检索

**基本流程：**

在事务阶段，客户端可以执行以下命令检索邮件：

1. **STAT命令**：获取邮件统计信息
   - 服务器响应："+OK {count} {total_size}"
   - 统计时排除已标记删除的邮件

2. **LIST命令**：获取邮件列表
   - 服务器响应邮件ID和大小列表
   - 格式："{id} {size}\n"
   - 以单独一行的"."结束

3. **RETR命令**：获取指定邮件内容
   - 格式：RETR {id}
   - 服务器响应邮件完整内容，包括From、To、Subject、Date等头部信息
   - 以单独一行的"."结束

**分支流程：**

1. 如果邮件ID不存在或已删除，服务器响应"-ERR No such message"
2. 如果命令格式错误，服务器响应"-ERR Unknown command"

**数据需求：**

1. 邮件ID
2. 邮件发件人
3. 邮件收件人
4. 邮件主题
5. 邮件正文
6. 邮件大小
7. 邮件创建时间

### 3.2.3 邮件删除

**基本流程：**

1. 客户端发送DELE命令标记邮件为删除
   - 格式：DELE {id}
   - 服务器响应"+OK Message {id} deleted"
   - 邮件被标记为待删除，但尚未真正删除

2. 客户端发送QUIT命令时，服务器执行实际删除操作
   - 将标记删除的邮件is_deleted字段设置为1
   - 服务器响应"+OK Bye"并关闭连接

**分支流程：**

1. 如果邮件ID不存在或已删除，服务器响应"-ERR No such message"
2. 如果客户端在QUIT前断开连接，标记删除的邮件不会被实际删除（POP3协议特性）

**数据需求：**

1. 邮件ID
2. 邮件删除标记状态

## 3.3 Web管理后台

Web管理后台提供基于浏览器的服务器管理界面，方便管理员进行系统配置和管理。

### 3.3.1 登录/注销

**基本流程：**

管理员通过浏览器访问Web管理后台（默认端口8080），进入登录界面。输入管理员账号和密码，系统验证后进入管理后台主界面。点击右上角"注销登录"可以退出登录。

**分支流程：**

1. 管理员登录信息为空时，系统会显示错误提示
2. 管理员登录账号与密码不匹配时，系统会显示错误提示
3. 管理员成功登录后，进入管理系统，点击"注销登录"，系统清除会话并返回登录界面

**数据需求：**

1. 管理员账户（邮箱格式）
2. 管理员密码（加密存储）

### 3.3.2 用户管理

**基本流程：**

管理员进入管理后台，点击侧边栏中的"用户管理"菜单。页面显示所有用户列表，包括用户名、角色（管理员/普通用户）、状态（激活/禁用）、创建时间等信息。管理员可以执行以下操作：

1. **创建用户**：填写用户名、密码，选择是否为管理员、是否激活，点击"创建用户"按钮
2. **编辑用户**：点击用户列表中的"编辑"按钮，修改用户信息（密码、角色、状态）
3. **删除用户**：点击用户列表中的"删除"按钮，确认后删除用户（不能删除自己）
4. **启用/禁用用户**：通过编辑功能修改用户状态

**分支流程：**

1. 创建用户时，如果用户名已存在，显示错误提示
2. 创建用户时，如果用户名格式不正确（不符合邮箱格式或域名不匹配），显示错误提示
3. 删除用户时，如果尝试删除自己，显示错误提示
4. 删除用户时，系统会确认操作，防止误删

**数据需求：**

1. 用户名（邮箱格式）
2. 密码（加密存储）
3. 用户角色（管理员/普通用户）
4. 用户状态（激活/禁用）
5. 用户创建时间
6. 用户邮箱容量限制
7. 用户当前邮箱使用量

### 3.3.3 服务管理

**基本流程：**

管理员进入管理后台，点击侧边栏中的"服务管理"菜单。页面显示SMTP和POP3服务的当前状态（运行中/已停止）、端口号、最后启动时间等信息。管理员可以执行以下操作：

1. **启动服务**：点击"启动"按钮启动SMTP或POP3服务
2. **停止服务**：点击"停止"按钮停止运行中的服务
3. **查看服务状态**：实时显示服务运行状态和端口监听情况
4. **修改端口**：通过系统设置页面修改服务端口（需要重启服务生效）

**分支流程：**

1. 启动服务时，如果端口已被占用，显示错误提示
2. 启动服务时，如果服务已在运行，显示提示信息
3. 停止服务时，系统会确认操作
4. 修改端口时，如果端口号无效（不在1-65535范围内），显示错误提示
5. 修改端口时，如果端口已被占用，显示错误提示

**数据需求：**

1. 服务名称（SMTP/POP3）
2. 服务状态（运行中/已停止）
3. 服务端口号
4. 服务进程ID（PID）
5. 服务最后启动时间
6. 服务最后停止时间

### 3.3.4 系统设置

**基本流程：**

管理员进入管理后台，点击侧边栏中的"系统设置"菜单。页面显示当前系统配置，包括：

1. **SMTP端口设置**：默认25，可修改
2. **POP3端口设置**：默认110，可修改
3. **服务器域名设置**：默认test.com，可修改
4. **管理员密码修改**：修改当前登录管理员密码

管理员修改设置后，点击"保存"按钮保存配置。

**分支流程：**

1. 修改端口时，如果端口号无效，显示错误提示
2. 修改端口后，需要重启对应服务才能生效
3. 修改密码时，需要输入旧密码和新密码，系统验证后更新
4. 修改域名后，新注册的用户需要使用新域名

**数据需求：**

1. SMTP端口号
2. POP3端口号
3. 服务器域名
4. 管理员密码（加密存储）

### 3.3.5 过滤规则管理

**基本流程：**

管理员进入管理后台，点击侧边栏中的"过滤规则"菜单。页面显示所有过滤规则列表，包括规则类型（邮箱/IP）、规则值、动作（阻止/允许）、状态（激活/禁用）、创建时间等信息。管理员可以执行以下操作：

1. **添加过滤规则**：选择规则类型（邮箱/IP），输入规则值，选择动作（阻止/允许），输入描述（可选），点击"添加"按钮
2. **删除过滤规则**：点击规则列表中的"删除"按钮，确认后删除规则
3. **启用/禁用规则**：通过编辑功能修改规则状态

**分支流程：**

1. 添加邮箱过滤规则时，如果邮箱格式不正确，显示错误提示
2. 添加IP过滤规则时，如果IP地址格式不正确，显示错误提示
3. 如果规则已存在，显示错误提示
4. 删除规则时，系统会确认操作

**数据需求：**

## 3.4 邮件传输模块

### 3.4.2 SMTP 协议的实现

SMTP（Simple Mail Transfer Protocol）协议是用于发送电子邮件的标准协议。本系统实现了SMTP服务器，能够接收客户端发送的邮件。

**实现架构：**

SMTP服务器采用PHP Socket编程实现，主要类为`SimpleSmtpServer`，位于`src/protocol/SmtpServer.php`。服务器采用单线程阻塞模式，每次处理一个客户端连接。

**核心实现流程：**

1. **服务器初始化**
   - 创建TCP Socket，绑定到指定端口（默认25）
   - 设置SO_REUSEADDR选项，允许端口重用
   - 监听客户端连接请求

2. **客户端连接处理**
   - 接受客户端连接，获取客户端IP地址
   - 记录连接日志到数据库

3. **SMTP协议交互**
   - **220响应**：服务器发送欢迎消息"220 mail.simple.com SMTP Ready"
   - **HELO/EHLO命令**：客户端发送问候命令，服务器验证IP过滤规则后响应"250 OK - Hello"
   - **MAIL FROM命令**：客户端指定发件人，服务器验证发件人过滤规则后响应"250 Sender OK"
   - **RCPT TO命令**：客户端指定收件人（支持多个），服务器验证收件人过滤规则和邮箱容量后响应"250 Recipient OK"
   - **DATA命令**：客户端请求发送邮件内容，服务器响应"354 Start mail input, end with <CRLF>.<CRLF>"
   - **邮件内容接收**：服务器逐行接收邮件内容，直到遇到单独一行的"."结束
   - **250响应**：邮件保存成功后响应"250 Mail accepted"
   - **QUIT命令**：客户端断开连接，服务器响应"221 Bye"

4. **邮件解析与存储**
   - 解析邮件头部，提取Subject字段
   - 解码MIME编码的主题（支持Base64和Quoted-Printable）
   - 提取邮件正文内容
   - 计算邮件大小（字节）
   - 将邮件信息存储到数据库emails表

5. **过滤机制**
   - IP地址过滤：在HELO命令后检查客户端IP是否在黑名单
   - 发件人过滤：在MAIL FROM命令后检查发件人邮箱是否被阻止
   - 收件人过滤：在RCPT TO命令后检查收件人邮箱是否被阻止

6. **邮箱容量管理**
   - 在RCPT TO阶段进行初步容量检查（估算50KB）
   - 接收完整邮件后使用实际大小进行最终检查
   - 如果邮箱已满，拒绝该邮件并记录日志

**关键技术点：**

- 使用PHP `socket_create()`创建TCP Socket
- 使用`socket_accept()`接受客户端连接
- 使用`socket_read()`和`socket_write()`进行数据读写
- 使用PDO预处理语句防止SQL注入
- 使用`password_verify()`验证用户密码
- 支持UTF-8编码，正确处理中文邮件

### 3.4.3 POP3 协议的实现

POP3（Post Office Protocol version 3）协议是用于接收电子邮件的标准协议。本系统实现了POP3服务器，允许客户端通过POP3协议检索和下载邮件。

**实现架构：**

POP3服务器采用PHP Socket编程实现，主要类为`SimplePop3Server`，位于`src/protocol/Pop3Server.php`。服务器采用状态机模式，包含三个状态：AUTH（认证）、TRANSACTION（事务）、UPDATE（更新）。

**核心实现流程：**

1. **服务器初始化**
   - 创建TCP Socket，绑定到指定端口（默认110）
   - 设置SO_REUSEADDR选项
   - 监听客户端连接请求

2. **客户端连接处理**
   - 接受客户端连接，获取客户端IP地址
   - 发送欢迎消息"+OK POP3 Simple Server Ready"
   - 进入AUTH状态

3. **用户认证阶段（AUTH）**
   - **USER命令**：客户端发送用户名，服务器验证用户是否存在且激活
     - 如果用户存在：响应"+OK User found"，保存当前用户
     - 如果用户不存在：响应"-ERR User not found"
   - **PASS命令**：客户端发送密码，服务器验证密码
     - 如果密码正确：加载用户邮件列表，响应"+OK Logged in, {count} messages"，进入TRANSACTION状态
     - 如果密码错误：响应"-ERR Invalid password"，重置用户状态
   - **QUIT命令**：客户端断开连接，响应"+OK Bye"

4. **事务阶段（TRANSACTION）**
   - **STAT命令**：返回邮件统计信息
     - 格式："+OK {count} {total_size}"
     - 排除已标记删除的邮件
   - **LIST命令**：返回邮件列表
     - 格式：每行"{id} {size}"，以单独一行的"."结束
     - 排除已标记删除的邮件
   - **RETR命令**：返回指定邮件内容
     - 格式：RETR {id}
     - 响应：邮件完整内容，包括From、To、Subject、Date等头部信息，以单独一行的"."结束
   - **DELE命令**：标记邮件为删除
     - 格式：DELE {id}
     - 响应："+OK Message {id} deleted"
     - 邮件被标记为待删除，但尚未真正删除
   - **QUIT命令**：执行实际删除操作，响应"+OK Bye"，进入UPDATE状态

5. **更新阶段（UPDATE）**
   - 将标记删除的邮件is_deleted字段设置为1
   - 重置用户状态和删除列表
   - 关闭连接

**关键技术点：**

- 状态机设计：AUTH -> TRANSACTION -> UPDATE
- 延迟删除机制：DELE命令只标记删除，QUIT时才真正删除
- 邮件列表缓存：登录时一次性加载用户所有邮件到内存
- 使用PDO预处理语句查询数据库
- 支持多行响应（LIST和RETR命令）

## 3.6 数据库

数据库是邮件服务器系统的核心数据存储层，采用MySQL关系型数据库管理系统。数据库设计遵循第三范式，保证数据的一致性和完整性。

### 3.6.1 概念数据模型

概念数据模型（CDM）描述了系统中实体及其之间的关系，不涉及具体的数据库实现细节。

**主要实体：**

1. **用户（User）**
   - 属性：用户ID、用户名、密码哈希、是否管理员、是否激活、创建时间
   - 关系：一个用户可以有多个邮件（作为发件人或收件人）

2. **邮件（Email）**
   - 属性：邮件ID、发件人、收件人、主题、正文、大小、是否删除、是否已读、创建时间
   - 关系：一封邮件属于一个发件人用户和一个收件人用户

3. **系统日志（ServerLog）**
   - 属性：日志ID、日志类型、消息内容、客户端IP、用户ID、创建时间
   - 关系：一条日志可能关联一个用户

4. **系统设置（SystemSetting）**
   - 属性：设置ID、设置键、设置值、更新时间
   - 关系：独立实体，无外键关系

5. **过滤规则（FilterRule）**
   - 属性：规则ID、规则类型、规则值、动作、描述、是否激活、创建时间
   - 关系：独立实体，无外键关系

6. **服务状态（ServiceStatus）**
   - 属性：服务ID、服务名称、是否运行、进程ID、最后启动时间、最后停止时间
   - 关系：独立实体，无外键关系

7. **邮箱限制（MailboxLimit）**
   - 属性：限制ID、用户ID、大小限制（字节）
   - 关系：一个邮箱限制属于一个用户

**实体关系图（ER图）描述：**

```
User (1) ──────< (N) Email (作为发件人)
User (1) ──────< (N) Email (作为收件人)
User (1) ──────< (0..1) MailboxLimit
User (0..1) ────< (N) ServerLog
```

### 3.6.2 物理数据模型

物理数据模型（PDM）描述了数据库的具体实现，包括表结构、字段类型、索引、外键约束等。

**数据库基本信息：**

- 数据库名称：mail_server
- 字符集：utf8mb4
- 排序规则：utf8mb4_unicode_ci
- 存储引擎：InnoDB（支持事务和外键约束）

**表结构设计原则：**

1. 所有表使用自增主键（AUTO_INCREMENT）
2. 时间字段使用TIMESTAMP类型，自动记录创建和更新时间
3. 布尔类型使用TINYINT(1)，0表示false，1表示true
4. 字符串字段根据实际需要设置长度，避免浪费空间
5. 外键约束使用ON DELETE SET NULL或ON DELETE CASCADE，保证数据完整性
6. 关键字段建立唯一索引，防止重复数据

### 3.6.9 filter_rules表

**表名：** filter_rules

**功能描述：** 过滤规则表，存储邮件和IP地址的过滤规则，用于阻止垃圾邮件和恶意访问。

**字段说明：**

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | INT | PRIMARY KEY, AUTO_INCREMENT | 规则唯一标识 |
| rule_type | ENUM('email', 'ip') | NOT NULL | 规则类型（邮箱/IP地址） |
| rule_value | VARCHAR(255) | NOT NULL | 规则值（邮箱地址或IP地址） |
| action | ENUM('block', 'allow') | DEFAULT 'block' | 动作（阻止/允许） |
| description | TEXT | NULL | 规则描述（可选） |
| is_active | TINYINT(1) | DEFAULT 1 | 是否激活（0=禁用，1=激活） |
| created_at | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | 创建时间 |

1  | email     | spam@example.com  | block  | 垃圾邮件    | 1         | 2024-01-01 10:00:00
2  | ip        | 192.168.1.200     | block  | 恶意IP      | 1         | 2024-01-01 10:05:00

**使用说明：**

- **邮箱过滤**：在SMTP服务器处理MAIL FROM和RCPT TO命令时检查
- **IP过滤**：在SMTP服务器处理HELO命令后检查客户端IP
- 支持阻止（block）和允许（allow）两种动作
- 可以启用或禁用规则，无需删除
- 规则类型和规则值的组合必须唯一
- Web管理界面提供规则的添加、删除、启用/禁用功能

**过滤逻辑：**

- 查询时按action降序排序，优先匹配阻止规则
- 如果找到匹配的阻止规则，直接拒绝
- 如果找到匹配的允许规则，允许通过
- 如果没有匹配规则，默认允许通过

**关联关系：**

- 独立实体，无外键关系

### 3.6.10 user_mailbox_limits表

**表名：** user_mailbox_limits

**功能描述：** 用户邮箱容量限制表，存储每个用户的邮箱大小限制，用于邮箱容量管理。

**字段说明：**

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | INT | PRIMARY KEY, AUTO_INCREMENT | 记录唯一标识 |
| user_id | INT | FOREIGN KEY, NOT NULL | 用户ID（关联users表） |
| size_limit_bytes | BIGINT | DEFAULT 104857600 | 邮箱大小限制（字节，默认100MB） |

**索引：**

- PRIMARY KEY (id)
- FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
- UNIQUE KEY unique_user (user_id)

**示例数据：**

```
id | user_id | size_limit_bytes
1  | 1       | 104857600  (100MB)
2  | 2       | 52428800   (50MB)
3  | 3       | 209715200  (200MB)
```

**使用说明：**

- 每个用户只能有一条邮箱限制记录（UNIQUE约束）
- 新用户创建时，如果没有设置限制，使用系统默认值（100MB）
- 使用MailboxRepository进行限制的查询和设置
- SMTP服务器在接收邮件时检查收件人邮箱容量
- 支持动态修改用户邮箱容量限制
- 删除用户时，关联的邮箱限制记录自动删除（CASCADE）

**容量计算：**

- 邮箱使用量 = 该用户所有未删除邮件的size_bytes总和
- 容量检查：使用量 + 新邮件大小 <= 容量限制
- 如果超出限制，拒绝接收邮件并返回"552 Mailbox full"错误

**关联关系：**

- 一对一：一个用户有一个邮箱容量限制

# 5. 项目总结

## 5.1 任务难点

在邮件服务器系统的开发过程中，遇到了以下几个主要难点：

1. **SMTP和POP3协议实现**
   - **难点**：需要深入理解SMTP和POP3协议的标准规范（RFC 5321和RFC 1939），正确实现协议交互流程
   - **解决方案**：仔细研读RFC文档，参考标准实现，使用telnet工具进行协议测试，逐步完善协议实现
   - **收获**：深入理解了邮件协议的工作原理，掌握了Socket网络编程技术

2. **多收件人邮件处理**
   - **难点**：SMTP协议支持一封邮件发送给多个收件人，需要正确处理多个RCPT TO命令
   - **解决方案**：使用循环接收多个RCPT TO命令，为每个收件人单独存储邮件记录，支持部分收件人失败的情况
   - **收获**：学会了处理复杂的协议交互场景

3. **邮箱容量管理**
   - **难点**：需要在接收邮件前检查收件人邮箱容量，但此时还不知道邮件的实际大小
   - **解决方案**：采用两阶段检查：RCPT TO阶段使用估算值（50KB）进行初步检查，接收完整邮件后使用实际大小进行最终检查
   - **收获**：学会了在不确定数据大小的情况下进行容量管理

4. **POP3延迟删除机制**
   - **难点**：POP3协议要求DELE命令只标记删除，QUIT时才真正删除，需要维护删除列表
   - **解决方案**：使用状态机设计，在TRANSACTION状态维护待删除邮件ID列表，QUIT时批量更新数据库
   - **收获**：理解了状态机设计模式的应用

5. **Web服务管理**
   - **难点**：通过Web界面启动和停止后台服务进程，需要获取进程ID并监控服务状态
   - **解决方案**：使用nohup命令后台运行PHP脚本，通过shell_exec获取进程ID，使用netstat/ss/lsof命令检查端口监听状态
   - **收获**：掌握了Linux进程管理和端口监控技术

6. **数据库设计兼容性**
   - **难点**：SMTP服务器使用字符串存储发件人/收件人，而系统需要支持用户ID关联
   - **解决方案**：同时保留sender/recipient（字符串）和sender_id/recipient_id（外键）字段，兼容两种存储方式
   - **收获**：学会了数据库设计的向后兼容性考虑

7. **字符编码处理**
   - **难点**：邮件主题和正文可能使用MIME编码（Base64或Quoted-Printable），需要正确解码
   - **解决方案**：实现MIME头解码函数，支持Base64和Quoted-Printable解码，统一转换为UTF-8编码
   - **收获**：深入理解了字符编码和MIME标准

8. **安全性实现**
   - **难点**：防止SQL注入、XSS攻击、密码泄露等安全问题
   - **解决方案**：使用PDO预处理语句防止SQL注入，使用htmlspecialchars防止XSS，使用password_hash加密密码，实现CSRF令牌保护
   - **收获**：掌握了Web应用安全防护的基本方法
